This page is the source of truth for **how bitHuman is built and shipped**. If you want to understand the moving parts before integrating, or you're an AI coding agent trying to map the ecosystem, start here.

## Install matrix

| Target                  | Command                                                                                                |
| ----------------------- | ------------------------------------------------------------------------------------------------------ |
| Python                  | `pip install bithuman` (1.12.4 вҖ” legacy batched API; streaming wheel queued)                           |
| Swift (iOS / macOS)     | SwiftPM `.package(url: "https://github.com/bithuman-product/bithuman-sdk", from: "1.17.0")`            |
| Android (Kotlin / Java) | Gradle `implementation("ai.bithuman:sdk:1.17.0")` (Maven Central)                                      |
| Rust (mac / Linux / Windows) | Cargo crate (not yet published) вҖ” built from the libessence-v1.17.0 source tag                    |
| CLI (Homebrew)          | `brew install bithuman-product/bithuman/bithuman` (1.17.0)                                             |
| CLI (curl-pipe)         | `curl -fsSL https://github.com/bithuman-product/bithuman-sdk/releases/latest/download/install.sh \| sh` |
| Flutter plugin          | `flutter pub add bithuman` (renamed from `bithuman_avatar` in v1.17.0; pub.dev publish queued)         |

The Swift, Kotlin, Rust, CLI, and engine flavors are produced from the same `libessence-v1.17.0` source tag and expose the streaming push/pull API (C ABI v6) introduced in v1.16.0. The Python wheel is still on `libessence-v1.12.4` (legacy batched API); the streaming wheel is queued behind the unified Rust + PyO3 build. The unified [install page](/getting-started/install) shows all channels side-by-side.

The reference apps ship in non-overlapping flavors: a [Flutter `bithuman` plugin](https://pub.dev/packages/bithuman) Essence app that covers macOS + iOS + Android (the easy cross-platform path); a native macOS Expression app at [`expression/mac/`](https://github.com/bithuman-product/bithuman-apps/tree/main/expression/mac) and a native iPadOS Expression app at [`expression/ipad/`](https://github.com/bithuman-product/bithuman-apps/tree/main/expression/ipad) (the on-device LLM/TTS path with MLX); and the Rust `bithuman` CLI for terminal-launched flows on macOS, Linux, and Windows. iPhone Expression remains in `archive/` while the per-app memory budget firms up вҖ” iPhone today is Essence-only.

### Layer 1 вҖ” `libessence` engine

A portable C++17 core (~12 k LoC) that turns 16 kHz mono PCM audio into 25 fps BGR avatar frames. The same source tree compiles to a macOS / iOS xcframework, an Android `libessence_jni.so`, a Linux/macOS Python module, the Rust CLI binary, and Linux x86_64 / aarch64 + Windows x86_64 dynamic libraries. The engine exposes a stable C ABI (`be_*` symbols, currently **ABI v6**) вҖ” every higher layer is a thin wrapper over it.

App developers never import this layer directly. It's documented for engine internals and parity-test consumers вҖ” see [`cpp/README.md`](https://github.com/bithuman-product/bithuman-sdk/blob/main/cpp/README.md).

#### Streaming API (C ABI v6)

ABI v6 introduced a streaming push/pull split that replaces the legacy "pass the entire accumulated audio buffer on every tick" pattern. The legacy shape was O(n) per tick in cumulative audio length and missed the 40 ms render budget after ~10 s of conversation; the new shape is O(n_new) on append and O(1) on frame pull, bit-exact vs the legacy batched API.

| Symbol                              | Cost     | Role                                          |
| ----------------------------------- | -------- | --------------------------------------------- |
| `be_runtime_push_audio(r, pcm, n)`  | O(n_new) | Append PCM, incrementally extend mel cache    |
| `be_runtime_pull_frame(r, hint, вҖҰ)` | O(1)     | Pull the next composed BGR frame              |
| `be_runtime_ticks_available(r)`     | O(1)     | How many composed frames are ready to drain   |
| `be_runtime_reset_stream(r)`        | O(1)     | End-of-utterance: drop cached audio + counters |

Surfaced today in the **Swift** binding (`Bithuman.Runtime.pushAudio` / `pullFrame` / `ticksAvailable` / `resetStream`), the **Kotlin** binding (`ai.bithuman.sdk.Runtime.pushAudio` / `pullFrame` / `ticksAvailable` / `resetStream`), and the **Rust** crate that backs the `bithuman` CLI. The Python wheel is still on the legacy batched API (`AsyncBithuman.push_audio` / `runtime.run()`); the streaming wheel is queued behind the unified Rust + PyO3 build.

Performance reference (single conversation, tight loop, CPU EP, one intra-op thread):

| Platform                       | Mean / tick | p99    | Sustained | Peak RSS |
| ------------------------------ | ----------- | ------ | --------- | -------- |
| Apple M5 (24 GB, macOS 26.3)   | 0.89 ms     | 1.0 ms | 1130 FPS  | 120 MB   |
| iPhone 17 Pro                  | 1.26 ms     | 1.4 ms | 793 FPS   | 59 MB    |
| Snapdragon 8 Gen 2 (Z Fold 5)  | 3.10 ms     | 3.6 ms | 322 FPS   | 139 MB   |

All three platforms run pixel-identical frames (max-abs diff вүӨ 18 LSB vs Python golden).

### Layer 2 вҖ” language bindings

Idiomatic bindings each language's developers can drop straight into their project:

| Binding | Coord                  | Platforms                        | Surface                          | Streaming API |
| ------- | ---------------------- | -------------------------------- | -------------------------------- | ------------- |
| Python  | `bithuman` on PyPI     | macOS arm64, Linux x86_64/aarch64 | pybind11 cdylib                 | legacy batched (1.12.4); streaming wheel queued |
| Swift   | SwiftPM `Bithuman` (`from: "1.17.0"`) | iOS device + sim, macOS arm64 | 3-slice xcframework, static link | yes (1.17.0) |
| Kotlin  | `ai.bithuman:sdk:1.17.0` (Maven Central) | Android `arm64-v8a`, minSdk 29 | JNI shim, static link | yes (1.17.0) |
| Rust    | Cargo crate (not yet published) | macOS arm64, Linux x86_64/aarch64, Windows x86_64 | crate, static link | yes (1.17.0) |

The public types are the same shape across bindings: an `Avatar` convenience facade for the single-conversation case, plus a `Fixture` + `Runtime` split (`Fixture.createRuntime()` / `Runtime(fixture)`) for multi-conversation hosts. Every binding speaks the same `.imx` model format and produces pixel-identical output. See the per-binding streaming pages: [Swift](/swift-sdk/streaming) В· [Kotlin](/kotlin-sdk/streaming).

### Layer 3 вҖ” end-user surfaces

Built on top of the bindings:

- **`bithuman` Rust CLI** вҖ” single Rust binary that links `libessence` statically. Three voice / text / avatar modes, each with a cloud (OpenAI Realtime) and on-device (`--local`) backend; `voice` and `text` auto-pick cloud when `OPENAI_API_KEY` is set, on-device otherwise. `bithuman avatar` (browser-served by default) gains a server-side `--openai` mode in v1.17.0 вҖ” host captures mic + plays speaker locally; the printed `http://127.0.0.1:вҖҰ` URL renders the lipsync view in any browser. Default OpenAI Realtime model is `gpt-realtime-mini` (cheapest tier). Same UX on macOS / Linux / Windows. Distributed via Homebrew (`brew install bithuman`) and a curl-pipe install script. Full reference: the [CLI page](/getting-started/cli) and [`docs/CLI.md`](https://github.com/bithuman-product/bithuman-sdk/blob/main/docs/CLI.md). Replaces the previous Swift `bithuman-cli` (retired in v1.17.0).
- **Flutter `bithuman` plugin** вҖ” one Dart codebase covering macOS + iOS + Android, wrapping the Swift and Kotlin bindings under a single `AvatarScreen` UI. macOS + iOS use a WebSocket OpenAI Realtime through the plugin's VP-IO `RealtimeAudioIO` (Apple hardware AEC); Android uses flutter_webrtc's libwebrtc native pipeline (libwebrtc native AEC). Settings sheet hot-updates the system prompt via `session.update`. Renamed from `bithuman_avatar` in v1.17.0.
- **Reference apps** вҖ” non-overlapping scopes:

  | Demo | Platforms | Runtime | LLM / TTS | Distribution |
  |---|---|---|---|---|
  | `flutter/bithuman/example/` | macOS В· iOS В· Android | Essence | Cloud (OpenAI Realtime) | `flutter run` |
  | `expression/mac/` | macOS only | Expression | On-device (MLX) or Cloud | Sparkle DMG / `swift run` |
  | `expression/ipad/` | iPadOS only | Expression | On-device (MLX) or Cloud | App Store / TestFlight |
  | `bithuman avatar` (Rust CLI) | macOS / Linux / Windows terminal | both Expression + Essence | both On-device + Cloud | `brew install bithuman` |

  See [bithuman-apps](https://github.com/bithuman-product/bithuman-apps) for the full reference-app catalogue. iPhone Expression is parked in `archive/` while the per-app memory budget firms up; iPhone today is Essence-only via the Flutter plugin.

## Model weights

Avatar `.imx` fixtures live in **proprietary cloud storage** and are downloaded at runtime from authenticated endpoints, then cached locally:

| Surface | Cache location | First-launch download |
|---|---|---|
| Python / Swift / Kotlin (Essence) | `~/.cache/bithuman/models/` | ~500 MB per `.imx` |
| Legacy on-device Expression (Swift) | `~/.cache/bithuman/expression/` | ~1.6 GB universal bundle |
| bithuman CLI (video / `--local` mode) | Same as above | ~5вҖ“7 GB (LLM + TTS + Expression) |

Weight downloads are SHA-256 verified and cached. Subsequent launches are instant.

## Authentication & billing

The same credential drives every surface, with two equivalent env-var names matching platform conventions:

```
BITHUMAN_API_SECRET    # Python SDK, Kotlin SDK, REST API, CLI
BITHUMAN_API_KEY       # Swift SDK (Apple-platform convention)
```

The SDK never holds the long-lived secret in process memory: on runtime initialisation it exchanges the secret for a short-lived **runtime token** at `https://api.bithuman.ai/v1/runtime-tokens/request`. Tokens auto-renew on a 1-request-per-minute heartbeat. Failed heartbeats trigger a 5-minute offline grace window before the avatar pauses.

Full flow + per-SDK setup: [/getting-started/authentication](/getting-started/authentication). Pricing: [/getting-started/pricing](/getting-started/pricing).

## Why this shape?

A few design constraints drove the architecture:

- **One engine, every platform.** The same C++ source produces the iOS xcframework, the Android `.so`, the Linux/macOS Python wheel, and the Rust CLI binary. Pixel-identical output across all targets вҖ” verified by golden-parity tests in CI.
- **Zero transitive deps for SwiftPM / Gradle / pip consumers.** Static link of `libessence.a` into each binding means an app's `Package.swift` / `build.gradle.kts` / `requirements.txt` only needs the bithuman dependency вҖ” no ONNX Runtime, no FFmpeg, no HDF5 to add manually.
- **Same credential for every surface.** `BITHUMAN_API_KEY` and `BITHUMAN_API_SECRET` are the same value; only the env-var name differs by platform convention.
- **Audio-only mode is unmetered.** Without an attached avatar the runtime is fully offline and bills nothing вҖ” encourages experimentation.
- **Model weights stay cloud-gated.** The proprietary IP is the model, not the glue. Code can be open-ish; weights stay locked.
