The streaming push/pull API on `ai.bithuman.sdk.Runtime` is the
recommended shape for any conversation that runs longer than a few
seconds. It lands in `ai.bithuman:sdk:1.16.0` (Maven Central) on top of
libessence C ABI v6, and is bit-exact vs the legacy batched API.

<Note>
**Why this exists.** The legacy batched API required passing the entire
accumulated audio buffer on every tick, which grew O(n) per tick in
cumulative audio length. Past ~10 s of conversation a single tick blew
the 40 ms render budget. The streaming API splits that into an O(n_new)
append (`pushAudio`) and an O(1) pull (`pullFrame`), so cost is flat
regardless of session length.
</Note>

## API surface

```kotlin
package ai.bithuman.sdk

class Runtime {
    /** Append PCM to the runtime's mel cache. O(n_new). */
    fun pushAudio(audioPCM: ShortArray, offset: Int = 0, length: Int = audioPCM.size)

    /** Pull the next composed BGR frame into the caller's buffer. O(1).
     *  `frameOut` must be sized to width * height * 3.
     *  `frameIdxHint = -1` lets the runtime advance internally. */
    fun pullFrame(frameOut: ByteArray, frameIdxHint: Int = -1)

    /** How many composed frames are ready to drain right now. */
    val ticksAvailable: Long

    /** Drop cached audio + per-stream counters at end of utterance. */
    fun resetStream()
}
```

The legacy methods (`Avatar.composeFromFile`, `Avatar.composeAsBitmaps`,
`Runtime.tick(...)`) continue to work unchanged for existing
integrations.

## Minimal driver

```kotlin
import ai.bithuman.sdk.Fixture
import ai.bithuman.sdk.Runtime

Fixture(modelPath).use { fx ->
    val rt = Runtime(fx)
    val info = fx.info()
    val frameOut = ByteArray(info.frameWidth * info.frameHeight * 3)

    // On every audio chunk arriving from AudioRecord / WebRTC sink:
    fun onAudio(pcm: ShortArray) {
        rt.pushAudio(pcm)

        // Drain composed frames at 25 fps.
        while (rt.ticksAvailable > 0) {
            rt.pullFrame(frameOut, frameIdxHint = -1)
            renderer.present(frameOut, info.frameWidth, info.frameHeight)
        }
    }

    // End of utterance: drop the audio cache so the next turn starts fresh.
    fun onEndOfTurn() = rt.resetStream()
}
```

`pushAudio` and `pullFrame` are independent — push as audio arrives,
pull on a 40 ms display tick. Back-pressure is the caller's choice:
either drain everything in `ticksAvailable` whenever new audio comes
in, or pace the pull on a `Choreographer` callback and let
`ticksAvailable` rise transiently.

## Threading

A single `Runtime` instance is **not** internally synchronised — pin
the push/pull pair to one thread (typically a dedicated background
worker), or wrap calls in your own mutex. Multi-conversation hosts
should still share **one `Fixture`** across many `Runtime` instances;
each runtime is independent.

## When to call `resetStream()`

Reset at the boundary between user turns: barge-in, end-of-utterance
detection, agent switch, or any time the cached audio history is no
longer relevant. Skipping the reset doesn't break correctness, but
keeps the mel cache growing across turns and trims the maximum
session length you can run before hitting the per-runtime audio
buffer cap.
