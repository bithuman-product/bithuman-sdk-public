`ai.bithuman:sdk` is an Android AAR that drops the bitHuman avatar
runtime into a Kotlin or Java app. Audio in (16 kHz mono PCM), 25 fps
BGR frames out. All inference runs on-device; the avatar pipeline is
metered through a 1-request-per-minute heartbeat to `api.bithuman.ai`.

This is the **Kotlin flavor of Layer 3** in the
[3-layer architecture](/getting-started/architecture) — a thin JNI shim
over the cross-platform [`libessence`](/getting-started/architecture#layer-1--libessence-engine) engine.

## Compatibility

| Field | Value |
| ----- | ----- |
| Maven coord | `ai.bithuman:sdk:1.16.0` (Maven Central) |
| ABI | `arm64-v8a` only |
| Min SDK | 29 (Android 10) |
| Compile SDK | 35 |
| NDK | 28.0.13004108 |
| libessence ABI | v6 (streaming push/pull) |
| AAR size | ~14 MB |

The AAR bundles three native libraries:

| Library | Size | Role |
| ------- | ---- | ---- |
| `libessence_jni.so` | ~11 MB | engine + JNI shim + OpenSSL 3.4 + libcurl 8.10 statically linked |
| `libonnxruntime.so` | ~27 MB | ONNX Runtime (CPU EP) |
| `libc++_shared.so` | ~1.2 MB | NDK C++ runtime |

Auth heartbeat is live (vendored OpenSSL + libcurl) — no stub mode. Set
`BITHUMAN_UNMETERED=1` for dev / parity-test runs against an audio-only
pipeline.

## Public API

All public types live in `ai.bithuman.sdk`:

| Tier | Types | Use when… |
| ---- | ----- | --------- |
| High-level | `Avatar`, `ComposedFrame` | Single conversation, one fixture in scope |
| Low-level | `Fixture`, `FixtureInfo`, `Runtime`, `ComposeResult` | Multi-conversation host — one fixture, many runtimes |
| Streaming | `Runtime.pushAudio` / `pullFrame` / `ticksAvailable` / `resetStream` | Long live conversations — see [streaming page](/kotlin-sdk/streaming) |
| Config | `ExecutionProvider` | EP hint (`CPU` / `AUTO` / `NNAPI` / `QNN`) |
| Errors | `BithumanException` | Single exception type with typed error codes |
| Auth | `BithumanAuth`, `AuthState` | Direct heartbeat control (rarely needed) |
| Version | `Fixture.libraryVersion()` / `Fixture.abiVersion()` | Linked libessence info |

The `Avatar` facade wraps `Fixture + Runtime` for the single-conversation
case. For multi-conversation hosting share one `Fixture` (~344 MB resident
on a typical 1248×704 model) across many `Runtime` instances (~36 MB
delta each) — at N=10 concurrent runtimes the memory efficiency is ~5.6×
vs naive per-conversation loads.

## Hardware floor

The library targets `arm64-v8a` Android devices on modern silicon.
Reference benchmark numbers from `bench/` on **Snapdragon 8 Gen 2**
(Galaxy Z Fold 5), Essence v1, CPU EP, intra-op threads = 1:

| Metric | Value |
| ------ | ----- |
| Tight-loop mean / tick | 3.96 ms |
| Sustained FPS | 252 |
| Peak RSS (PSS) | 139 MB |

That's ~10× headroom over the 25 fps / 40 ms tick budget. Older arm64
chips (e.g. SD865-era) have not been characterized; treat them as
unsupported for production until measured.

## Authentication

```
BITHUMAN_API_SECRET    # Android env / Application.onCreate System.setProperty
```

Sign in at [www.bithuman.ai → Developer → API Keys](https://www.bithuman.ai/#developer)
to fetch a secret. The library never holds the long-lived secret —
it exchanges it for a short-lived runtime token on first compose, then
auto-renews on a 1-request-per-minute heartbeat (5-min offline grace).
Full flow: [Authentication](/getting-started/authentication).

## ABI roadmap

- `arm64-v8a` — shipped today.
- `armeabi-v7a`, `x86_64` — deferred. The C++ core builds for both but
  the AAR pipeline + ONNX Runtime prebuilts are wired only for arm64.
