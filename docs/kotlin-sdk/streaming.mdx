The streaming push/pull API on `ai.bithuman.sdk.Runtime` is the right shape for conversations that run longer than a few seconds, or for any integration where audio arrives incrementally (microphone, WebRTC sink, TTS stream).

Each push is constant cost; each pull returns one composed frame. Session length doesn't degrade per-frame performance.

## API surface

```kotlin
package ai.bithuman.sdk

class Runtime {
    /** Append PCM to the runtime. */
    fun pushAudio(audioPCM: ShortArray, offset: Int = 0, length: Int = audioPCM.size)

    /** Pull the next composed frame into the caller's buffer.
     *  `frameOut` must be sized to width * height * 3 (packed BGR).
     *  `frameIdxHint = -1` lets the runtime advance internally. */
    fun pullFrame(frameOut: ByteArray, frameIdxHint: Int = -1)

    /** How many composed frames are ready to drain right now. */
    val ticksAvailable: Long

    /** Drop cached audio at end of utterance. */
    fun resetStream()
}
```

## Minimal driver

```kotlin
import ai.bithuman.sdk.Fixture
import ai.bithuman.sdk.Runtime

Fixture(modelPath).use { fx ->
    val rt = Runtime(fx)
    val info = fx.info()
    val frameOut = ByteArray(info.frameWidth * info.frameHeight * 3)

    // On every audio chunk arriving from AudioRecord / WebRTC sink:
    fun onAudio(pcm: ShortArray) {
        rt.pushAudio(pcm)

        while (rt.ticksAvailable > 0) {
            rt.pullFrame(frameOut, frameIdxHint = -1)
            renderer.present(frameOut, info.frameWidth, info.frameHeight)
        }
    }

    // End of utterance: drop the audio cache so the next turn starts fresh.
    fun onEndOfTurn() = rt.resetStream()
}
```

`pushAudio` and `pullFrame` are independent — push as audio arrives, pull on a 40 ms display tick. Back-pressure is your choice: drain everything in `ticksAvailable` whenever new audio comes in, or pace pulls on a `Choreographer` callback.

## Threading

A single `Runtime` instance is **not** internally synchronized — pin the push/pull pair to one thread (typically a dedicated background worker), or wrap calls in your own mutex.

Multi-conversation hosts should still share **one `Fixture`** across many `Runtime` instances; each runtime is independent.

## When to call `resetStream()`

Reset at boundaries between user turns: barge-in, end-of-utterance detection, agent switch, or any time the cached audio history is no longer relevant.
